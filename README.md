## 📖 Overview

Cloud infrastructure has a real, measurable carbon footprint — but most teams have no visibility into it. **Cloud Carbon Footprint Analyzer** lets you define your cloud architecture (manually, from a sample, or synced from AWS Cost Explorer), estimates the CO₂ emissions generated by your compute, storage, and network resources, and recommends greener regions, cleaner providers, and concrete efficiency improvements.

Built to bridge **FinOps** and **sustainability**, the tool turns resource specs into actionable, provider-agnostic carbon insights — no cloud credentials required to try it.

---

## ✨ Features

- **📤 Architecture definition** — Define resources through a guided form, load a realistic sample architecture, or sync usage from AWS Cost Explorer (falls back to illustrative demo data if no AWS credentials are configured).
- **📊 CO₂ emissions estimation** — Calculates estimated monthly energy use and CO₂ output per resource (compute, database, storage, network), using power-draw coefficients and PUE by provider.
- **🌍 Greener region suggestions** — Ranks every other region on the same provider by projected CO₂ reduction if the architecture were relocated there.
- **⚖️ Cross-provider comparison** — Compares AWS, Azure, and GCP for the same architecture deployed to the closest equivalent geographic region on each.
- **⚡ Energy-efficient recommendations** — Rule-based suggestions: rightsizing under-utilized instances, carbon-aware scheduling for bursty workloads, storage tiering for large SSD volumes, and network optimization for high-egress resources — each with an estimated CO₂ savings figure.
- **📈 Interactive dashboard** — Stat cards, a per-resource emissions chart, and the app's signature carbon-intensity gauge used consistently across every view.

## 🚀 Getting Started

### Option A — Docker Compose (recommended)

```bash
git clone https://github.com/archanavijayan0112-spec/cloud-carbon-footprint-analyzer.git
cd cloud-carbon-footprint-analyzer
docker-compose up --build
```

- Frontend: `http://localhost:3000`
- Backend API: `http://localhost:8000`
- API docs (Swagger): `http://localhost:8000/docs`

### Option B — Manual setup

**Backend**
```bash
cd backend
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # SQLite works out of the box for local dev — set DATABASE_URL=sqlite:///./app.db
uvicorn app.main:app --reload
```

**Frontend**
```bash
cd frontend
npm install
npm run dev
```

Open `http://localhost:3000`, click **Load sample architecture**, then **Analyze footprint** to see the full flow without needing any cloud credentials.

### Optional: live AWS sync

Set `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` in `backend/.env` with Cost Explorer read access, then use **Sync from AWS** in the upload screen. Without credentials, the app transparently falls back to illustrative demo usage data so the rest of the product remains fully explorable.

## 🧮 How Emissions Are Estimated

```
energy_kwh = resource_power_draw(kWh) × PUE(provider)
co2_kg     = energy_kwh × grid_intensity(gCO₂/kWh) / 1000
```

- **Compute/database** — power draw is interpolated between each instance family's min/max watts-per-vCPU based on assumed (or specified) utilization, multiplied by vCPU count and hours per month.
- **Storage** — flat watts-per-GB-month coefficients differentiated by SSD / HDD / object storage.
- **Network** — flat kWh-per-GB-transferred coefficient.
- **Grid intensity** — static per-region gCO₂/kWh values for AWS, Azure, and GCP (`backend/app/data/grid_intensity.json`), loosely modeled on the methodology used by the open-source Cloud Carbon Footprint project. These are illustrative averages — swap in a live source like Electricity Maps for production accuracy.

## 🧪 Running Tests

```bash
cd backend
pytest tests/ -v
```

20 tests cover: energy scaling with utilization/vCPUs/storage size, CO₂ scaling with grid intensity, unknown-region error handling, architecture-level aggregation, region-suggestion ranking and filtering, cross-provider comparison correctness (including non-mutation of input resources), and each recommendation rule.
